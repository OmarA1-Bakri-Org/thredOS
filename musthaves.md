
## The V.1 target

thredOS V.1 should do six things well:

1. install and run a pack
2. compile a pack into a sequence + surfaces
3. execute steps inside surface-scoped runs
4. enforce deterministic gates in code
5. keep append-only traces and audit records
6. export a run bundle for third-party evaluation

It should **not** try to do internal benchmark scoring, leaderboard logic, cryptographic sealing, or “world-class eval” in V.1.

---

## 1) Canonical thredOS V.1 structure

Build on the current local `.threados/` runtime shape and formalize it into this tree. Current docs already make `.threados/sequence.yaml`, `.threados/prompts/`, `.threados/runs/<runId>/<stepId>/`, `.threados/state/`, and `.threados/policy.yaml` the local runtime truth. ([GitHub][2])

```text
.threados/
  sequence.yaml                 # user-authored sequence instance
  policy.yaml                   # user-authored execution policy

  packs/
    <packId>/
      <version>/
        pack.yaml
        phases/
        agents/
        surfaces/
        gates/
        schemas/
        policies/
        templates/
        exports/

  prompts/
    <promptId>.md

  surfaces/
    <surfaceId>/
      surface.json
      access.json
      barrier.json
      state.json

  runs/
    <runId>/
      run.json
      trace.ndjson
      approvals.ndjson
      gate-decisions.ndjson
      surfaces/
        <surfaceId>/
          input.manifest.json
          compiled-prompt.md
          artifact.manifest.json
          outputs/
          logs/
          private/

  state/
    sequence-state.json
    surface-state.json
    run-state.json
    gate-state.json
    indexes.json

  audit/
    actions.ndjson
    policy.ndjson

  exports/
    <runId>/
      bundle.json
      trace.ndjson
      approvals.ndjson
      gate-decisions.ndjson
      artifacts/
```

### What is editable vs generated

Only these should be human-edited:

* `sequence.yaml`
* `policy.yaml`
* pack files under `packs/...`
* prompt files under `prompts/`

Everything else should be generated by thredOS. That cuts drift and prevents agents from directly mutating canonical runtime state.

## 2) Core domain contracts

This is the minimum contract set I would make canonical in V.1.

### Sequence

The orchestration graph.

Required fields:

* `id`
* `name`
* `pack_id`
* `pack_version`
* `thread_type`
* `steps[]`
* `deps[]`
* `default_policy_ref`
* `created_at`
* `updated_at`

### Step

One executable node.

Required fields:

* `id`
* `name`
* `phase`
* `kind` (`base | parallel | chained | fusion | baton | long-autonomy`)
* `agent_ref`
* `prompt_ref`
* `surface_ref`
* `depends_on[]`
* `input_contract_ref`
* `output_contract_ref`
* `gate_set_ref`
* `completion_contract`
* `side_effect_class`

### Surface

The runtime container for execution and inspection.

Required fields:

* `id`
* `name`
* `surface_class`
* `parent_surface_id`
* `workspace_root`
* `artifact_root`
* `visibility`
* `allowed_read_scopes[]`
* `allowed_write_scopes[]`
* `reveal_policy`
* `metadata_budget`
* `isolation_label`
* `status`

### Run

One attempt.

Required fields:

* `id`
* `sequence_id`
* `step_id`
* `surface_id`
* `attempt`
* `status`
* `executor`
* `model`
* `policy_snapshot_hash`
* `compiled_prompt_hash`
* `input_manifest_ref`
* `artifact_manifest_ref`
* `started_at`
* `ended_at`
* `timing_summary`
* `cost_summary`

### GateDecision

Pure-code decision output.

Required fields:

* `id`
* `subject_type`
* `subject_ref`
* `gate_type`
* `status`
* `reason_codes[]`
* `evidence_refs[]`
* `decided_by` (`threados`)
* `decided_at`

### Approval

Human or policy approval record.

Required fields:

* `id`
* `action_type`
* `target_ref`
* `requested_by`
* `status`
* `approved_by`
* `approved_at`
* `notes`

### TraceEvent

Append-only event log entry.

Required fields:

* `ts`
* `run_id`
* `surface_id`
* `actor`
* `event_type`
* `payload_ref`
* `policy_ref`

### BarrierAttestation

Honest isolation record.

Required fields:

* `surface_id`
* `run_id`
* `isolation_label`
* `cross_surface_reads_denied`
* `shared_semantic_projection`
* `reveal_state`
* `contamination_events[]`

## 3) Surface model for V.1

This is the most important part.

### Surface classes

I would ship four surface classes:

* `shared`
* `private`
* `sealed`
* `control`

#### shared

Normal working surface. Readable where dependencies allow it.

#### private

Readable only by steps bound to that surface and by control-plane metadata views.

#### sealed

Like `private`, but **no semantic projection** outside the surface before reveal. Other surfaces and the control plane can only see non-semantic manifests and gate proofs.

#### control

Used for orchestration, approvals, and gate status. It should not read sealed semantic payloads pre-reveal.

### Isolation labels

Do not use vague words. Make the claim explicit.

I would ship these labels:

* `NONE`
* `THREADOS_SCOPED`
* `HOST_ENFORCED`  *(reserved for later if you add OS/container ACLs)*

For V.1, I would expect most sealed flows to be labeled **`THREADOS_SCOPED`**, not `HOST_ENFORCED`.

### Barrier rules for V.1

For a `sealed` surface:

* no cross-surface reads through UI, CLI, or API
* no semantic previews in inspectors
* no shared logs containing semantic payload
* only `input.manifest.json`, `artifact.manifest.json`, and gate proofs visible externally
* reveal must be an explicit state transition
* every reveal writes a `BarrierAttestation`

This is enough to make the claim honest for V.1 without pretending you have cryptographic or container-level isolation.

## 4) Deterministic gate engine

This needs to be a native thredOS subsystem. The LLM can propose work; it does **not** decide pass/fail.

Current thredOS already centers deterministic CLI control, runtime execution, local state mutation, gate management, and runtime-event emission, so this should be added as a first-class code path rather than left in prompt prose. ([GitHub][2])

### Gate types for V.1

I would implement only these deterministic gate types:

* `deps_satisfied`
* `required_inputs_present`
* `policy_pass`
* `output_schema_pass`
* `artifact_manifest_pass`
* `approval_present`
* `surface_access_pass`
* `reveal_allowed`
* `completion_contract_pass`

### Gate output behavior

Every gate decision should emit:

* status: `PASS | BLOCK | NEEDS_APPROVAL`
* reason codes like `DEP_MISSING`, `SCHEMA_INVALID`, `POLICY_BLOCKED`, `REVEAL_LOCKED`
* evidence refs into manifests, traces, or approvals

No free-text “manager judgment” in the decision layer.

## 5) Policy structure for V.1

Current policy already supports SAFE/POWER mode, command allowlists, cwd patterns, fanout/concurrency caps, and forbidden regexes, with SAFE requiring confirmation and chat proposing rather than auto-executing. Keep that, but extend it slightly for this use case. ([GitHub][3])

### Keep the current fields

* `mode`
* `command_allowlist`
* `cwd_patterns`
* `max_fanout`
* `max_concurrent`
* `forbidden_patterns`

### Add these V.1 fields

* `side_effect_mode: manual_only | approved_only | free`
* `network_mode: off | allowlist | open`
* `allowed_domains: []`
* `surface_default_visibility`
* `cross_surface_reads: deny | dependency_only`
* `sealed_surface_projection: manifest_only`
* `export_mode: off | local_bundle`

That is enough for the SaaS factory use case without ballooning the policy engine.

## 7) Pack model for the SaaS factory

This is how I would encode the factory itself inside thredOS.

```text
packs/
  saas-factory/
    1.0.0/
      pack.yaml
      policies/
        default.yaml
      agents/
        orchestrator.md
        ideator.md
        validator.md
        selector.md
        builder.md
        commercial.md
      surfaces/
        control.surface.yaml
        shared-review.surface.yaml
        sealed-track.surface.yaml
      phases/
        00_intake.yaml
        10_extraction.yaml
        20_parallel_ideation.yaml
        30_validation.yaml
        40_selection.yaml
        50_build.yaml
        60_packaging.yaml
        70_final_rank.yaml
      gates/
        core.gates.yaml
        reveal.gates.yaml
        build.gates.yaml
      schemas/
        step-output.schema.json
        gate-decision.schema.json
        surface.schema.json
        run.schema.json
        export-bundle.schema.json
      templates/
        briefs/
        summaries/
        manifests/
      exports/
        third-party-eval-bundle.yaml
```

### `pack.yaml` should define

* pack identity and version
* supported thread types
* default policy
* available agents
* surface classes
* phase order
* dynamic surface rules
* gate sets
* export bundle schema

### Key pack behavior

The pack compiler should be able to take something like:

* source asset
* number of parallel tracks
* policy mode
* selected models

and produce:

* `sequence.yaml`
* instantiated surfaces
* step bindings
* prompt refs
* gate refs

That is how the giant prompt becomes a thredOS-native pack instead of remaining a monolith.

## 8) Third-party eval-ready export

Since you want evals out of core V.1, make export **excellent**.

Every completed run should be exportable as a bundle with:

* pack ID + version
* sequence snapshot
* policy snapshot
* surface definitions
* runs
* gate decisions
* approvals
* traces
* artifact manifests
* cost/timing summaries

Example:

```json
{
  "bundle_version": "1.0",
  "pack": { "id": "saas-factory", "version": "1.0.0" },
  "sequence_snapshot": "...",
  "policy_snapshot": "...",
  "surfaces": ["..."],
  "runs": ["..."],
  "gate_decisions": ["..."],
  "approvals": ["..."],
  "trace_files": ["trace.ndjson"],
  "artifact_manifests": ["..."],
  "timing_summary": "...",
  "cost_summary": "..."
}
```

That gives you a clean V.2 bridge without slowing V.1 down.

## 9) UI minimum for V.1

The current architecture already has the workbench shell, canvas, inspector/forms, chat panel, selected thread-surface/run state, and a Thread Runner gate component. So I would keep the UI changes narrow: add inspector tabs instead of inventing a second product surface. ([GitHub][2])

### Add only these UI pieces

* **Surface tab** — class, visibility, reveal state, isolation label
* **Gate tab** — deterministic decisions + reason codes
* **Trace tab** — append-only event stream
* **Approvals drawer** — pending/approved side effects and reveals
* **Export action** — generate bundle for external eval
* **Thread Runner** — visible but marked “external eval integration / V.2”

That is enough for V.1.

## 10) Non-goals for V.1

I would explicitly keep these out:

* internal benchmark scoring
* reliability dashboards
* contamination harnesses
* cryptographic sealing / zk-style proof files
* multi-tenant cloud orchestration
* autonomous outbound side effects
* semantic grading by the orchestrator

Those are all valid later, but not now.

## 11) Build order

This is the order I would build it in.

### Milestone 1 — Contracts and filesystem

Lock the canonical objects and `.threados/` tree.

**Done means:** pack install works, sequence compiles, state folders created, schemas validated.

### Milestone 2 — Surface-scoped execution

Every step runs inside a bound surface and writes surface-scoped artifacts.

**Done means:** no run can write outside its assigned surface paths through normal thredOS execution.

### Milestone 3 — Deterministic gates

Pure-code gate engine with reason codes.

**Done means:** pass/block decisions never depend on model prose.

### Milestone 4 — Barrier/reveal model

`sealed` surfaces, manifest-only projection, explicit reveal.

**Done means:** sealed surface semantics are not visible in UI/API before reveal.

### Milestone 5 — Pack compiler

SaaS factory pack compiles into a runnable sequence + surfaces.

**Done means:** one command or one UI action instantiates the workflow.

### Milestone 6 — Export bundle

Third-party eval export.

**Done means:** complete run can be bundled and handed off externally.

---

## 12) The three highest-value things to build first

If you want maximum leverage with minimum V.1 scope, build these first:

1. **Canonical contracts + `.threados/` tree**
2. **Deterministic gate engine**
3. **Surface-scoped traces + export bundle**

Those three pieces turn thredOS from “agentic workbench” into “credible runtime substrate.”

The first thing to lock is the canonical contracts and `.threados/` tree. Once that is fixed, the rest becomes implementation rather than debate.
